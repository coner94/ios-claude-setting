## When to Activate

- 코드 리뷰가 필요하거나 요청받았을 때
- 사용자가 `/code-review`로 직접 호출할 때

## Step 1: 변경 범위 파악

- 변경된 파일 목록과 diff를 분석한다
- 변경의 목적과 의도를 파악한다
- 영향받는 레이어와 모듈을 확인한다

## Step 2: 코드 분석

각 변경 사항을 아래 관점에서 검토한다:

- **정확성**: 로직이 의도대로 동작하는가
- **안전성**: 옵셔널 처리, 에러 핸들링, 동시성 안전성
- **단순성**: 더 짧게 쓸 수 있는 코드가 없는가, 불필요한 추상화가 없는가
- **컨벤션**: `.claude/rules/swift-style.md` 규칙 준수
- **성능**: 불필요한 연산, 메인 스레드 블로킹
- **테스트**: 변경에 대한 테스트가 충분한가

## Step 3: 리뷰 결과 작성

```markdown
# Code Review: {PR 제목 또는 변경 요약}

## 요약
- 전체적인 코드 품질 평가 (1~2줄)
- 승인 여부: Approve / Request Changes / Comment

## Critical (반드시 수정)
- [{파일명}:{라인}] {문제 설명}
  - 원인: {왜 문제인지}
  - 제안: {구체적 수정 방안}

## Warning (수정 권장)
- [{파일명}:{라인}] {문제 설명}
  - 제안: {개선 방안}

## Suggestion (선택적 개선)
- [{파일명}:{라인}] {제안 내용}

## 긍정적 피드백
- {잘 작성된 부분에 대한 코멘트}
```

---

## Rules

- **코드를 직접 수정하지 않는다.** 리뷰 의견만 제시한다.
- 심각도를 Critical / Warning / Suggestion으로 명확히 구분한다.
- 문제 지적 시 반드시 구체적인 파일명과 라인을 명시한다.
- 단순 스타일 지적보다 로직/안전성/단순성 문제를 우선한다.
- 잘 작성된 코드에 대한 긍정적 피드백도 포함한다.
- Swift 6 Concurrency 관련 데이터 레이스 위험을 식별한다.
